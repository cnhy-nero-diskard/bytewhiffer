## Context

Three independent rendering defects, all confirmed by hands-on use rather than inferred from code alone:

1. The block hover tooltip (`treemap_panel`, `app.rs`) renders `hit.trail.join("/")` as a single unbounded-width label. `PopupAnchor::Pointer` repositions the popup to stay inside the viewport; near an edge, the available width collapses, and since path segments have no spaces to break on, egui hard-wraps mid-word into an illegible single-character-per-line column.
2. On a dense `C:\` scan, the same tooltip visibly lags far behind a smoothly-moving cursor. `draw_children` re-sorts every visible node's children and, for every card-eligible block, re-tessellates a blurred `Shadow::as_shape` plus a hand-built gradient mesh (`paint_card`) — unconditionally, every frame, regardless of whether the tree changed. `hover_pos()` reflects the freshest input available when the frame starts, so this isn't a stale-input bug; it's that a slow frame takes long enough to build and present that, by the time it reaches the screen, the cursor has moved measurably further — and the denser the view, the slower every frame, so the perceived lag scales with density. This exact risk was flagged, and left with an unimplemented fallback plan, in the archived `soft-elevation-theming` design.
3. Toolbar buttons/path field/breadcrumb chips (`chrome_button`, `chrome_chip`, `paint_surface`) reuse `theme::card_shadow()`'s constants (`SHADOW_BLUR: 7`, `SHADOW_OFFSET: [0,2]`) verbatim. Those were tuned by eye against treemap cards, which by `MIN_CARD_SIDE`'s own floor rarely render smaller than ~40-50px in practice (below that, blocks tend to fall to the flat fallback or get absorbed into dense clusters). Chrome buttons sit at ~26-28px tall — right at the size where the same absolute blur radius has no room to fade gently, so it reads as a second, harder-edged rectangle rather than a subtle lift. The theming spec already calls for chrome to use "the same gradient/shadow/radius treatment as treemap blocks, scaled appropriately" — that clause was never actually implemented; `paint_surface` passes rects straight through with no size-aware adjustment.

## Goals / Non-Goals

**Goals:**
- Make the tooltip legible regardless of where on screen it's anchored.
- Make the tooltip track the pointer responsively on dense scans, without regressing the visual richness (shadow/gradient cards) that sparse/typical scans already have.
- Make chrome elevation actually read as a subtle lift rather than a doubled shape, at the sizes chrome elements actually render at.
- Keep `scanner/` and `treemap.rs` free of any rendering concerns, per the existing module boundary (`CLAUDE.md`).

**Non-Goals:**
- A general retained-mode/dirty-rect rendering architecture for the treemap. egui redraws the full frame every paint regardless of what changed; the fix here is reducing *what* gets computed and tessellated per frame for dense scenes, not building a scene-graph cache.
- Changing `MIN_CARD_SIDE`, the flat-fallback rendering itself, or any color/hue selection logic — none of that is implicated by these three defects.
- A continuous/formula-based shadow-scaling system that scales smoothly across arbitrary element sizes. Chrome elements occupy a narrow, known size band (~26-34px tall); treemap cards rarely render near that band in practice. Two fixed, hand-tuned presets cover the actual cases; a general formula would be solving for sizes that don't occur.

## Decisions

**Tooltip text: pre-truncate with middle-ellipsis, don't rely on wrap behavior at all.** Disabling wrap alone just trades "wraps illegibly" for "overflows the popup frame" — it doesn't bound the tooltip's size. Instead, cap the displayed trail string to a fixed character budget (exact number tuned visually at implementation time) with a middle-ellipsis (e.g. `SteamLibrary/steamapps/…/pakchunk70-WindowsNoEditor.pak`), matching how Windows Explorer shortens long paths, then render that single bounded string with wrapping off. The full, untruncated path remains available in the persistent bottom status bar (`status_bar`, already unelided), so no information is lost — just not duplicated in the space-constrained popup.
- *Alternative considered*: set an explicit fixed pixel wrap width on the tooltip label instead of truncating the string. Rejected because it still produces a tall multi-line tooltip for long paths (just with cleaner line breaks) rather than the compact single-line hint a hover tooltip should be, and multi-word-per-line wrapping of a slash-delimited path is arguably less scannable than a single elided line.

**Tooltip lag: extend the existing size-based flat-fallback pattern to also be density-aware, rather than introducing a per-frame result cache.** The dominant per-frame cost is shadow/gradient tessellation (per the archived soft-elevation design's own risk analysis), not the `sort_by` call — sorting a node's children is a cheap integer comparison over a handful-to-low-hundreds of items. The already-documented, never-implemented fallback (drop blur for a hard-edged shadow) is the right lever, but applying it by a *global* visible-count check is simpler and safer than deciding per-block mid-frame (which would let some cards stay elevated while sibling cards silently go flat within the same view, reading as a rendering glitch rather than an intentional density response).
  - Compute a "how many entries are in play under the current focus" count once, when the focus path or `tree_rev` changes (not every frame) — this is already a natural cache-invalidation point the app tracks. Cache that count alongside the focus/tree state.
  - When that count exceeds a threshold (tuned at implementation time via `--debug-perf` against a scene shaped like a dense `C:\` scan, alongside the existing synthetic benches), `draw_children` renders every card-eligible block for that frame using the flat-fallback path (or a hard-edged, `blur: 0` shadow if a lighter middle ground reads better than fully flat) instead of the full blurred-shadow-and-gradient card — for the whole view, consistently, until the count drops back below the threshold.
  - *Alternative considered*: cache the fully painted `Shape`/`Mesh` output of `draw_children` (and the accompanying hit-rects) keyed on `(tree_rev, focus, viewport rect)`, and simply re-submit cloned shapes on frames where none of those changed. This is the more general fix and would preserve full visual richness at any density, but it's materially more invasive (introduces a display-list cache and a second, hit-test-only fast path) for a defect that the project's own prior design work already anticipated and pre-approved a simpler lever for. Left as a fallback if the density-threshold approach, measured with `--debug-perf`, doesn't close the gap enough on real dense trees.
  - *Alternative considered*: reorder the frame so hit-testing/tooltip positioning happens before painting. Rejected — `hover_pos()` already reflects the freshest input available when the frame starts regardless of where in the frame it's read; the lag comes from total frame *duration*, not from reading position too late within an already-slow frame.

**Chrome shadow: a second, smaller fixed preset for chrome, not a formula.** Add a chrome-scale shadow (smaller blur/offset, tuned by eye the same way the existing block-scale constants were) alongside the existing `card_shadow()`, and have `paint_surface` (the chrome-only entry point — `paint_card` for treemap blocks is untouched) use the chrome preset instead of the block preset. This directly fulfills the theming spec's existing "scaled appropriately" clause with a concrete, testable implementation.
- *Alternative considered*: scale blur/offset proportionally to `rect`'s size via a formula. Rejected as solving for a range of sizes that doesn't occur in practice — chrome elements cluster tightly around one size band, treemap cards rarely render near that band, so two hand-tuned presets are simpler and match the codebase's existing style of tuned-by-eye constants (`SHADOW_BLUR`, `GRAD_LIGHTEN`, etc. are all already hand-picked, not derived).

## Risks / Trade-offs

- **[Risk] The density threshold for the tooltip-lag fix is a single global switch per frame, so a view could visibly "flatten" all at once when crossing the threshold (e.g. drilling one level deeper crosses back under it)** → **Mitigation**: this is the same trade-off the existing `MIN_CARD_SIDE` per-block threshold already makes (a block crossing that size threshold on resize snaps between flat and elevated, per the theming spec's own "no intermediate or broken state" scenario) — consistent with an existing, accepted pattern rather than a new kind of visual discontinuity.
- **[Risk] Two hand-tuned shadow presets (block vs. chrome) could drift out of sync stylistically over time if one is retuned without the other** → **Mitigation**: both live in `theme.rs` next to each other with a shared doc comment cross-referencing the other, same as how `MIN_CARD_SIDE`/`MIN_NEST_AREA`/`MIN_NEST_SIDE` already cross-reference each other today.
- **[Trade-off] Middle-ellipsis truncation means the tooltip alone can no longer show a very long full path** → accepted, since the full path is already shown, unelided, in the persistent status bar the instant a block is hovered — the tooltip's job is a quick at-a-glance hint, not the authoritative full-path readout.

## Open Questions

- Exact truncation character budget for the tooltip, and the exact visible-entry-count threshold for the density-aware fallback, are implementation-time tuning against the real running app (via `--debug-screenshot*` and `--debug-perf`), not architectural decisions — consistent with how the archived soft-elevation design left its own thresholds open.
- Whether the density-aware fallback should render fully flat or use a `blur: 0` hard-edged shadow as its "dense" tier is also left for visual comparison at implementation time; both are cheap relative to the blurred default, and the choice is aesthetic rather than architectural.
